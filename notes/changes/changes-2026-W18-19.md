## E2E Playwright Push — Coverage From ~5 % to ~75-78 %

Before W18 : `src/app/` ~62 %, with `wip ~8 %`, `bw ~34 %`, several modules at 0 %. After W19 : `app/` ~75-78 %, `wip ~72 %`, `bw ~63 %`, measured live against the dev server via the new `flask-coverage` extension.

The push is structured in layers ; each unblocks the next.

**Layer 1 — Data-bearing profiles + cookie-aware POST**

- `find_data_bearing_profile` scans CSV profiles to pick one that actually owns items in the listing under test (the first profile of a community often has zero). Module-level cached.
- **Infra bug** : Playwright's `page.request.post()` does **not** carry the `BrowserContext` cookies — all authenticated POSTs were bouncing silently on `/auth/login` (status 200, body = login form) without an assertion on final URL, so 5 tests were green false positives. Replaced by `authed_post` / `authed_get` (via `page.evaluate("fetch(...)")`, which shares the browser cookie jar) + uniform `/auth/login not in resp["url"]` assertion.

**Layer 2 — Mail debug harness**

- New `app.flask.mail_debug` extension swaps `MAIL_BACKEND` to an in-memory `EmailBackend` and exposes `/debug/mail/{,messages,messages/<idx>,reset}`. Fail-closed : refuses to register outside `app.debug` or `FLASK_MAIL_DEBUG_PASSWORD`. Same shape as `flask-coverage`.
- Bypasses anti-spam counters when `MAIL_DEBUG_ACTIVE` (`email_limiter.is_email_sending_allowed`, `avis_matching.partition_by_cap`, `publication_notification_service._dispatch`). Without these bypasses, notify tests pass once then capture zero mails on the next run.
- Pytest fixture `mail_outbox` (function-scoped, auto-reset).

**Layer 3 — Mono-user mail-aware tests**

Tests that POST and verify captured mail :

**Layer 4 — Multi-user lifecycle tests**

Single Playwright `page` switching between 2 users (login → action → logout → login → action). Reaches chains no mono-user test can.

- `test_avis_rdv_full_lifecycle` (5 steps, 5 mails : opportunity-OUI → propose → accept → confirm → cancel). Covers `/rdv-accept`, `/rdv-confirm` with `ACCEPTED`, `/rdv-cancel` with future `date_rdv`.

## Heroku R14/R15 — Memory Fix

Symptom : `Process running mem=800M (156%)` then `1029M (201%)` on Heroku web dyno (quota 512 MB).

Diagnosis on `src/server/main.py` :

- 3 Flask / Starlette apps mounted into a single Granian process (Flask aipress24 + sqladmin + POC). The POC imports its own blueprint tree — ~150-200 MB RSS for free.
- `gunicorn.conf.py` is dead code : the Procfile launches `python -m server`, not gunicorn.
- No explicit `workers=` passed to Granian.

Fix : POC mounted only when `POC_ENABLED=true` (import moved inside the conditional branch) ; `WORKERS = config("WEB_CONCURRENCY", cast=int, default=1)` passed to `granian.Granian(...)`.

Remaining paths noted : move scheduler-thread out of web dyno into a worker dyno ; `index_all` loads `cls.query.all()` per Typesense collection without `yield_per` / `expunge_all` ; consider jemalloc buildpack.

## Firefox E2E Flakiness — Vite HMR Sockets

Symptom : under Firefox + a loaded suite (>20 cumulated tests), `page.goto(...)` timed out on routes that curl reached in 130 ms direct but >90 s in browser.

Diagnosis : the HTML embeds Vite dev-server `<script src=...>` (HMR client + ES-module graph). With many successive Playwright contexts, Firefox serialises new scripts behind earlier HMR sockets → DCL stalls indefinitely.

Fix : autouse fixture `_abort_vite_dev_assets` in `e2e_playwright/conftest.py` routes `**://localhost:3000/**` → abort at the Playwright level. Pages still render their HTML, just without HMR (tests don't use it). Verified : a test that hung > 180 s now passes in 12 s.

## Pyrefly Type-Error Cleanup — 300+ → 0

`uv run pyrefly check src` was reporting 300+ errors. Triage : the vast majority are SQLAlchemy descriptor false positives — `Mapped[T]` on instances resolves to `InstrumentedAttribute[T]` instead of `T`.

Approach :

- Category-level silencing in `[tool.pyrefly.errors]` for the most pervasive false-positive codes (`bad-argument-type`, `bad-index`, `no-matching-overload`, …).
- Per-call-site `# pyrefly: ignore[<code>]` inserted by `pyrefly suppress` for residual cases (scoped to the precise check). Self-documenting at the source.
- Stub gaps fixed by migration (enums — see next section) or suppressed (`singledispatch.register` decorator form, 5 cases in `services/social_graph/_adapters.py`).

**Real bugs surfaced during triage** :

- `app/flask/lib/templates.py:71` — `case Path(template_path):` : `pathlib.Path` has no `__match_args__`, so the pattern raised `TypeError: Path() accepts 0 positional sub-patterns (1 given)` at runtime whenever `self.template` was a `Path`. Fixed to `case Path() as template_path:` + `str(template_path)`.
- `app/modules/kyc/xls_parser.py:189` — `contact_type` could be unbound (loop without prior init). Latent `UnboundLocalError` if the first cell was empty. Initialised to `""` before the loop.
- `app/modules/kyc/field_label.py:42-50` — `get_ontology_content()` returns `list | dict`, but two callers indexed it with a string key. Tightened with `assert isinstance(onto_dict, dict)`.
- `app/modules/swork/components/base.py:164` — `sorted({selector(obj) ...})` with `selector` typed `object` (via `getattr`) broke the `sorted` specialisation. Annotated the comprehension as `set[Any]`.
- `app/services/sessions/_service.py:58` — `SessionService.get()` now annotated `-> Any`. The default `_marker = object()` had caused callers to infer `object`, breaking `if sub_id in id_key` in `kyc/views.py:369` and several other sites — single annotation fixes all of them.

Final state : `pyrefly check src` → `0 errors (129 suppressed, 70 warnings not shown)`.
