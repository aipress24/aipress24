## Bugs Surfaced By The Coverage Push (W18, 5 same-day fixes)

Each fix lands with its own regression test pinning post-fix
behaviour.

- **`notifications-mark-read-open-redirect`** (`f445f087`) — **P1
  security**. `POST /notifications/<id>/read` redirected blindly
  to the `url` form param (`_safe_next_url` was only applied to
  `next`). Refactored `_safe_next_url(form_key=...)`, applied to
  both. Phishing vector closed.
- **`wire-item-non-numeric-id-500`** (`f712e887`) — `get_obj`
  called `int(id)` or `base62.decode(id)` without catching
  `ValueError`. `/wire/<non-numeric>` → 500. Wrapped in
  `try/except (ValueError, TypeError) → raise NotFound from None`.
- **`kyc-validation-empty-session-500`** (`3aaceb01`) —
  `/kyc/validation` GET without `profile_id` in session crashed in
  `get_survey_profile("")`. Early-redirect to `/kyc/profile`.
- **`bw-configure-gallery-500`** (`20119d80`) —
  `stage_b1b.configure_gallery` didn't wrap the S3 upload block
  in a `try/except` like `stage_b1`. Any S3 hiccup → user-visible
  500. Try/except + flash error.
- **`wip-content-form-date-mismatch`** (`ffac7e47`) —
  Article/Sujet/Commande have 7 `DateTimeField`s without
  `validators=[InputRequired()]` but `Mapped[datetime]` (NOT
  NULL) columns — partial POST → 500 `IntegrityError`. Validators
  added directly in `_forms.py` (the generator
  `scripts/generate-forms3.py` is now obsolete — doesn't cover

## Heroku R14/R15 — Memory Fix

Commit `7e6cfe06`. Prod symptom : `Process running mem=800M (156%)`
then `1029M (201%)` on the Heroku web dyno (quota 512 MB).

Diagnosis on `src/server/main.py` :

- 3 Flask/Starlette apps mounted into a single Granian process
  (Flask aipress24 + sqladmin + POC). The POC imports its own
  blueprint tree — ~150-200 MB RSS for free.
- `gunicorn.conf.py` is dead code : the Procfile launches
  `python -m server`, not gunicorn.
- No explicit `workers=` passed to Granian — we were relying on the
  default.

Fix :

- POC mounted only when `POC_ENABLED=true`. The
  `from poc.app import …` import is moved inside the conditional
  branch so the sub-tree isn't even imported in prod.
- `WORKERS = config("WEB_CONCURRENCY", cast=int, default=1)`
  explicitly passed to `granian.Granian(...)`. Heroku → 1 fixed
  worker.

Remaining paths noted in the diag : (a) move the scheduler-thread
out of the web dyno into a worker dyno (each tick re-instantiates a
Flask via `scheduler.index()` — slow leak), (b) `index_all` loads
`cls.query.all()` per Typesense collection without `yield_per` or
`expunge_all`, (c) add the jemalloc buildpack.

## E2E Harness — `scripts/run_e2e.py`

Python script that spawns the server with the right options and
runs the tests with the right options.

- Spawns `make run` (with `COVERAGE_PROCESS_START` pointing at
  `pyproject.toml` for subprocess coverage), polls `/auth/login`
  until the server answers, then launches `pytest`.
- Auto-skips `--reruns` / `--timeout` flags if the plugins aren't
  installed (probes via `uv run python -c "import X"`).
- **Timestamped logs** in `local-notes/e2e-logs/<YYYYMMDD-HHMMSS>/` :
  `server.log` (file-only — doesn't pollute the console) +
  `pytest.log` (mirror of the pytest output). Implemented with a
  `_Tee` daemon thread per piped child, `bufsize=1` +
  `PYTHONUNBUFFERED=1` for live streaming.
- `--no-server` reuses an already-running dev server ; refuses to
  start if port 5000 is busy without the flag (anti-foot-gun).
- `--` forwards args verbatim to pytest (e.g.
  `uv run scripts/run_e2e.py e2e_playwright/wire/ -- -k purchase`).
- SIGTERM/SIGKILL cleanup of the whole process group (honcho + vite
  + backend) on normal exit, error, or KeyboardInterrupt.

## Firefox E2E Flakiness — Vite HMR Sockets

Symptom : under Firefox + a loaded suite (>20 cumulated tests),
`page.goto(...)` timed out on routes that curl reached in 130 ms
direct but >90 s in browser.

Diagnostic :

1. The server route answers in <200 ms (verified curl direct
   authenticated).
2. The HTML points to 6 synchronous `<script src=...>`, including
   2 toward the Vite dev server
   (`http://localhost:3000/@vite/client`, `/main.js`).
3. Vite serves a recursive ES-module graph — each `import` is a
   separate HTTP request + a WebSocket HMR per page.
4. With >20 successive Playwright contexts, the Firefox scheduler
   serialises new scripts behind earlier HMR sockets → DCL stalls
   indefinitely.

Fix : new autouse fixture `_abort_vite_dev_assets` in
`e2e_playwright/conftest.py` that routes `**://localhost:3000/**` →
abort at the Playwright level. Pages still render their HTML, just
without HMR (which the tests don't use). Sub-resources that were
blocked now fail-fast → DCL fires immediately. Verified : a test
that hung > 180 s now passes in 12 s.

## Pyrefly Type-Error Cleanup — 300+ → 0

`uv run pyrefly check src` was reporting 300+ errors. Triage : the
vast majority are SQLAlchemy descriptor false positives — `Mapped[T]`
on instances resolves to `InstrumentedAttribute[T]` instead of `T`.

Approach :

- Category-level silencing in `[tool.pyrefly.errors]` for the most
  pervasive false-positive codes (`bad-argument-type`, `bad-index`,
  `no-matching-overload`, …).
- Per-call-site `# pyrefly: ignore[<code>]` inserted by
  `pyrefly suppress` for the residual cases (scoped to the precise
  check — not a blanket disable). Now self-documenting at the
  source.
- Stub gaps fixed by migration (enums — see next section) or
  suppressed (`singledispatch.register` decorator form, 5 cases
  in `services/social_graph/_adapters.py`).

**Real bugs surfaced** during the triage :

- `app/flask/lib/templates.py:71` — `case Path(template_path):` :
  `pathlib.Path` has no `__match_args__`, so the pattern raised
  `TypeError: Path() accepts 0 positional sub-patterns (1 given)`
  at runtime whenever `self.template` was a `Path`. Verified
  reproducible in REPL. Fixed to
  `case Path() as template_path:` + `str(template_path)`.
- `app/modules/kyc/xls_parser.py:189` — `contact_type` could be
  unbound (loop without prior initialisation, unlike the
  `community = ""` pattern just above). Latent `UnboundLocalError`
  if the first cell was empty. Initialised to `""` before the
  loop, matching the `community` style.
- `app/modules/kyc/field_label.py:42-50` — `get_ontology_content()`
  returns `list | dict`, but two callers indexed it with a string
  key (docstring : "The ontology is a dict"). Tightened with
  `assert isinstance(onto_dict, dict)` after the call.
- `app/modules/swork/components/base.py:164` —
  `sorted({selector(obj) ...})` with `selector` typed `object`
  (via `getattr`) broke the `sorted` specialisation. Annotated
  the comprehension as `set[Any]`.
- `app/services/sessions/_service.py:58` —
  `SessionService.get()` now annotated `-> Any`. The default
  `_marker = object()` had caused callers to infer `object`,
  breaking `if sub_id in id_key` in `kyc/views.py:369` and several
  other sites — single annotation fixes all of them.

Final state : `pyrefly check src` → `0 errors (129 suppressed, 70
warnings not shown)`. Each suppression is per-line and labelled
with the exact error code so future stub improvements can be swept
back out.

## `aenum` → stdlib `enum` Migration

yields the lowercase name in both

## Admin — Grant / Revoke Admin From `/admin/show_user/<uid>`

New action surface on the existing admin user-detail page.

- `admin/views/show_user.py` : 2 new cases in `ShowUserView.post()`
  dispatch — `grant_admin` / `revoke_admin`. Helpers
  `_grant_admin` / `_revoke_admin` use
  `app.services.roles.add_role(user, RoleEnum.ADMIN)` /
  `user.remove_role(RoleEnum.ADMIN)`, idempotent (no-op if the
  desired state is already reached). `loguru.info` audit line with
  actor + target (id + email) on each transition.
- Self-revocation guard : `_revoke_admin` silently refuses to
  remove ADMIN from `current_user` (prevents admin lockout). The
  guard lives in the helper rather than in `post()` for
  testability — no need to go through an HTTP `test_request_context`
  that would commit and break test-transaction isolation.
- `admin/templates/admin/pages/show_user.j2` : conditional button
  next to "Désactiver" / "Exclure de l'organisation" — "Donner les
  droits admin" (indigo) when `not user.has_role("ADMIN")` ;
  "Retirer les droits admin" (red) when `user.has_role("ADMIN")`
  and `user.id != current_user.id`. Confirmation modals mirror the
  existing `confirm_deactivate_user` / `confirm_remove_org`
  patterns (DOM, Tailwind classes, HTMX `hx-post` +
  `HX-Redirect`).
- Tests : 5 new in `tests/b_integration/admin/test_show_user.py`
  calling the helpers directly under `app.test_request_context()`
  + `login_user(admin_user)` (same pattern as the existing
  `_deactivate_profile` / `_remove_organisation` tests) — grant
  adds the role, grant idempotent, revoke removes the role, revoke
  idempotent, self-revoke blocked. 7/7 passing.

UX choice validated via `AskUserQuestion` : self-revoke
**blocked** (vs. allowed-with-extra-confirmation or
unconditional). The button is hidden in the UI AND refused
server-side — defence in depth.
