# Deploying AIPress24 on Hop3

[`hop3.toml`](../hop3.toml) (repo root) drives the deployment. The app is built
natively (Hop3's Python toolchain runs `uv sync --no-dev` from `uv.lock`; the
frontend is served from the committed `vite/dist`) and served by gunicorn, with
the dramatiq-pg worker + scheduler as background processes and a `postgres`
addon that injects `DATABASE_URL`.

Two deployment modes are supported from the **same** `hop3.toml`. They differ
only in the config you apply before deploying.

## Secrets model (this repo is public)

No secret is ever committed. Non-secret structure lives in `hop3.toml` and
`etc/settings.toml`; everything else is applied with `hop3 config set`:

- **Real secrets** → `scripts/hop3-config.example.sh`. Copy it to
  `scripts/hop3-config.local.sh` (gitignored), fill in the `CHANGE_ME` values,
  and run it. The values currently in the old piku scripts (`setup.sh`,
  `setup_s3.sh`, `update-stripe-keys.sh`) belong here — and should be **rotated**,
  since they were committed in plaintext.
- **CI / blank** → `scripts/hop3-ci-config.sh` (safe to commit; no secrets).

## Mode 1 — Real deploy (production data)

```bash
# 1. Provision/select the app's postgres addon, then restore your dump into it
#    (out of band — e.g. `hop3 pg:credentials aipress24` then psql < dump.sql).
# 2. Apply real config + secrets:
cp scripts/hop3-config.example.sh scripts/hop3-config.local.sh
$EDITOR scripts/hop3-config.local.sh        # fill in CHANGE_ME (and rotate!)
sh scripts/hop3-config.local.sh
# 3. Deploy. before-run detects the populated DB (alembic_version present) and
#    runs `flask db upgrade` only (FLASK_SEED_BLANK unset), so your restored
#    data is never touched — only pending migrations apply.
hop3 deploy aipress24
hop3 app restart aipress24
```

## Mode 2 — Blank deploy (CI / testing)

```bash
# Fresh empty postgres addon, no real secrets needed.
sh scripts/hop3-ci-config.sh aipress24      # FLASK_SEED_BLANK=true, host check off
hop3 deploy aipress24
```

`before-run` (`scripts/hop3-before-run.sh` → `scripts/hop3_db_bootstrap.py`)
brings the schema to head whatever the DB state:

- **Empty DB** (blank/CI): `db.create_all()` (build the schema from the models)
  then stamp Alembic at head. A plain `flask db upgrade` can **not** build this
  app's schema from empty — the `recreate_baseline_for_existing_database`
  migration assumes the core tables already exist (they were originally made by
  `create_all`), so replaying from empty dies on `relation "aut_user" does not
  exist`.
- **Populated DB** (restored dump): `flask db upgrade` applies pending migrations.
- **Tables but no `alembic_version`**: refuse (unmanaged schema).

Then, because `FLASK_SEED_BLANK` is set, it runs `flask bootstrap` (reference
data: roles, taxonomies, countries, zip codes — fetched from the public
aipress24-data repo) and `flask data bootstrap-users` (the demo logins in
`users/*.yaml`). All idempotent, so redeploys are safe.

> Note: the `users/*.yaml` demo fixtures are currently stale (they carry a
> top-level `departement`, now a computed property on `User`), so
> `bootstrap-users` logs per-user import errors and creates no logins. This is
> tolerated (the deploy still succeeds with reference data seeded); fix the
> fixtures if CI needs to log in.

## What must be true to boot

`create_app()` hard-requires only: `DATABASE_URL` (addon), the four `S3_*`
settings, `SECRET_KEY`, and (for auth) `SECURITY_PASSWORD_SALT`. The last two are
generated by Hop3 on first deploy (`FLASK_SECRET_KEY` / `FLASK_SECURITY_PASSWORD_SALT`
in `hop3.toml [env]`, generated-once), so you never set them by hand. In the
default (development) Dynaconf environment used by CI, `etc/settings.toml` also
supplies dummy MinIO `S3_*` and a file mail backend, so a blank deploy boots with
no real secrets. Stripe, Sentry, Typesense and Mail all degrade gracefully
(warnings only) when unset.

## Operations

```bash
hop3 app logs aipress24          # runtime logs (web + workers)
hop3 app build-logs aipress24    # build logs
hop3 config show aipress24       # effective env (verify DATABASE_URL is present)
hop3 ps scale aipress24 web=2    # scale the web process
```
